# 1.3 Booking Confirmation Page

## Scope
- The screen-level specification for the Booking Confirmation Page displayed to the **client** after a booking reaches `CONFIRMED` status.
- This is a **client-only** page. Companions do not have this view.
- Covers: information displayed, available actions, day-of-service reveal (QR/PIN), and the page lifecycle until session start.
- This document will be refined in a future iteration.

### 1.3.1 Overview
* After the booking transitions from `PENDING` → `CONFIRMED` (see 1.2.1.6), the client's app automatically transitions to the **Booking Confirmation Page**.
* This page serves as the client's primary interface between booking confirmation and session start.
* The page remains active until the session is officially started via QR scan or PIN entry.

### 1.3.2 Information Display

#### 1.3.2.1 Status Update
* App UI transitions to the active booking view once the status becomes `CONFIRMED`.

#### 1.3.2.2 API
* **[GET] /bookings/{id}/details** — fetches core booking data (ID, Venue, Timing).

#### 1.3.2.3 Visible Data
* Booking ID
* Venue Name
* Venue Type (Mall / Club / Restaurant)
* Date & Time Slot
* **No companion-specific details** (names/photos) are shown at this stage.
* UI should focus on a **"Luxury Confirmation"** aesthetic.

### 1.3.3 Available Actions

#### 1.3.3.1 Cancel Booking
* **Primary Action:** "Cancel Booking" button.
* **Logic:** Tapping cancel sends a request to the backend. The system auto-calculates the refund/penalty based on the timeframe defined in 1.2.5.1.
* **Redirection:** Post-cancellation, the user is redirected back to the beginning of the "Hire Companion" flow (Step 1.2.1.1).

### 1.3.4 Day-of-Service Reveal

#### 1.3.4.1 Handshake Readiness
* On the day of service, the UI reveals:
    * **QR Code:** For the Duo to scan.
    * **Unique 4-Digit PIN:** For manual verbal verification if scanning fails.
    * **Meeting Point:** Decided during the client-companion matching phase (see 2.2.1). The client shares their location or sets a meeting point on the matching page. Companions can also contact the client via in-app notification as a fallback.

### 1.3.5 Page Lifecycle
* This view remains active until the session is officially started via QR scan or PIN entry.
* If the booking is cancelled, the page is dismissed and the client is redirected to the booking flow entry (1.2.1.1).
