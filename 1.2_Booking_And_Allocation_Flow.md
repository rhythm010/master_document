## Scope
- Booking process from the moment user opens the booking page
- Allocation process for companions
- Booking success flows
- Booking cancellation flows / rebooking flows

### 1.2 The Booking & Allocation Flow
#### 1.2.1 Functional (Client)

##### 1.2.1.1 Entry & Account Verification
*   **1.2.1.1.1 Flow:** Client opens the app and navigates to the "Hire Companion" section.
*   **1.2.1.1.2 Request:** **FE Check:** The system checks the client's current status.
    *   **Logic:** If a client has *any* active or future booking (Status: `PENDING`, `CONFIRMED`, `ACTIVE`), they are **automatically redirected** to the **Booking Details Page** for that booking. They cannot access the "New Booking" flow.
*   **1.2.1.1.3 Logic:** A single client account is strictly serialized to one booking at a time.
*   **1.2.1.1.4 Pending Flow Re-entry:** (Covered by 1.2.1.1.2 - Automatic Redirect).

##### 1.2.1.2 Venue Selection
*   **1.2.1.2.1 Request:** **[GET] /malls** to fetch partnered locations.
*   **1.2.1.2.2 Geolocation:** System requests GPS; FE suggests nearest mall at the top of the dropdown. **Global Visibility:** All partner malls are listed regardless of distance/country.
*   **1.2.1.2.3 Logic:** Bookings are specific to a mall and cannot be transferred. Changes require cancellation (subject to the 7h window).

##### 1.2.1.3 Scheduling & Availability
*   **1.2.1.3.1 Input:** Client selects a **Date** and a **Start Time**.
    *   **Booking Window:** 
        *   **Minimum Lead Time:** 24 Hours.
        *   **Maximum Advance:** 14 Days (2 Weeks).
*   **1.2.1.3.2 Source:** Data is derived from the **BE Roster Logic**.
*   **1.2.1.3.3 Logic:**
    *   **Fixed Duration:** All bookings are for a **Fixed 2-Hour Duration**. Users cannot customize the duration (e.g., select 3 hours).
    *   **Operating Hours:** The system automatically filters out slots that exceed the specific Mall's operating hours.
    *   **Capacity Only:** Returns valid slots based on the aggregate count of available Duos (Capacity) mapped to the venue.
    *   **Anonymity:** No specific Duo/Companion identity is exposed or assigned at this stage.
    *   **Static Data:** No auto-refresh logic is applied to this view.

##### 1.2.1.4 Review & Mandatory Consent
*   **1.2.1.4.1 Summary:** FE displays a detailed summary:
    *   **Context:** Mall, Date, Time.
    *   **Financials:** Full Price Breakdown (Base Rate + VAT + Service Fees = **Grand Total**).
    *   **Promo Codes:** Not supported in this version.
*   **1.2.1.4.2 Navigation:**
    *   **Edit:** User must navigate "Back" to modify Date/Time/Mall.
    *   **Timer:** No countdown timer or soft-lock on the slot at this stage.
*   **1.2.1.4.3 Action:** Clicking "Book Now" triggers the **Mandatory Consent Modal**.
*   **1.2.1.4.4 Consent Modal Content:**
    *   **Visuals:** A list of "Do Not" items (e.g., No Touch, No Harassment) displayed with illustrative images/icons.
    *   **Mechanism:** User must tick a single checkbox: **"I agree to all terms and conditions"**.
    *   **Enforcement:** The "Confirm & Pay" button is disabled until the checkbox is ticked.

##### 1.2.1.5 Booking Initiation & Payment Simulation (Phase 1)
*   **1.2.1.5.1 Request:** **[POST] /bookings** (MallID, Slot, Nickname).
*   **1.2.1.5.2 BE Operations:** Create record with `status: PENDING`. Apply 15-minute soft-lock on the Duo. Generate unique Booking ID.
*   **1.2.1.5.3 FE Simulation:** Phase 1 uses **Mock Data** for payments. 
    *   **UI:** Display a "Simulating Payment Authorization" spinner (3-5 seconds).
    *   **Logic:** System mocks a successful "Payment Hold" for the full estimated amount.
*   **1.2.1.5.4 Pending View Actions:** While in the `PENDING` state, the user is presented with a simplified dashboard. The only available action is a "Cancel Booking" button, which terminates the `PENDING` state and releases the soft-lock.

##### 1.2.1.6 Allocation & Confirmation Sync
*   **1.2.1.6.1 FE Polling:** FE polls **[GET] /bookings/{id}/status** (background auto-retries enabled) or listens for Realtime update.
*   **1.2.1.6.2 Success Path:** 
    *   **Logic:** Allocation engine finalizes the Duo match.
    *   **Transition:** The "Processing" page automatically converts into the **Booking Confirmation Page** without a redirect.
    *   **Notification:** Push notification sent: "Booking Confirmed".
*   **1.2.1.6.3 Failure Path:**
    *   **Logic:** If allocation fails (timeout or race condition), the system **Voids the Mock Transaction**.
    *   **UI Feedback:** The page displays an error message: "Booking cannot be completed. Slot no longer available."
    *   **Retry:** User must restart the flow to select a different slot.

##### 1.2.1.7 Final Confirmation & Information Display
*   **1.2.1.7.1 Status Update:** App UI transitions to the active booking view once the status becomes `CONFIRMED`.
*   **1.2.1.7.2 Request:** **[GET] /bookings/{id}/details** to fetch core booking data (ID, Mall, Timing).
*   **1.2.1.7.3 Visibility & UI:** Displays Booking ID, Mall name, and Date/Time Slot. No companion-specific details (names/photos) are shown at this stage. UI should focus on a "Luxury Confirmation" aesthetic.
*   **1.2.1.7.4 Actions:** 
    *   **Primary Action:** "Cancel Booking" button.
    *   **Logic:** Tapping cancel sends a request to the backend. The system auto-calculates the refund/penalty based on the timeframe defined in 1.2.5.1.
    *   **Redirection:** Post-cancellation, the user is redirected back to the beginning of the "Hire Companion" flow (Step 1.2.1.1).
*   **1.2.1.7.5 Handshake Readiness:** On the day of service, the UI reveals:
    *   **QR Code:** For the Duo to scan.
    *   **Unique 4-Digit PIN:** For manual verbal verification if scanning fails.
    *   **Meeting Point:** If GPS is denied, displays a specific location within the mall.
*   **1.2.1.7.6 Scope:** This view remains active until the session is officially started via QR scan or PIN entry.

##### 1.2.1.8 Session Maintenance & Termination
*   **1.2.1.8.1 Cancellation:** Clients can cancel up to 7 hours before the start time. Re-booking requires starting the flow fresh.
*   **1.2.1.8.2 Identity:** The system uses the client's profile identity. The client can set a nickname which will be visible to the companions.
*   **1.2.1.8.3 Admin Logic:** Admin has the authority to cancel a client's booking (triggering a notification). Admin cannot "update" a client booking, only cancel.

##### 1.2.1.9 Accessibility & Localization
*   **1.2.1.9.1 Standards:** Initial implementation follows standard OS accessibility settings; no bespoke luxury-tier accessibility features are planned for Phase 1.
*   **1.2.1.9.2 Localization:** App UI and AI chat agent support English and Arabic as per user profile.

#### 1.2.2 Functional (Companion) 
*   **1.2.2.1 Calendar & Blocking:** 
    *   **Logic:** Upon booking confirmation, the system **Auto-Assigns** the booking to the Duo. There is **no option to Decline** or Accept.
    *   **Block Duration:** The system blocks the Companion's calendar for **2 Hours (Session)** + **20 Minutes (Rest Buffer)**.
    *   **Notification:** The Companion does **NOT** receive an immediate notification of the booking.
    *   **Visibility:** Booking details remain hidden until **X Hours** (Configurable) before the start time.
    *   **Travel Time:** The Roster Algorithm handles geospatial constraints and travel buffers dynamically. Companions are expected to be present at the assigned location based on their Roster.
*   **1.2.2.2 Preparation Alerts:** A "Battery Check" push notification is sent a few hours before the booking start time to ensure companions have ample time to reach 100% charge.
*   **1.2.2.3 Shift Context & Cancellation:**
    *   **Context:** Companions operate in **Shifts** (e.g., 6 hours), within which multiple **Bookings** (e.g., 2 hours each) are assigned.
    *   **Cancellation Unit:** Companions can only cancel their **Entire Shift**. Partial cancellations (e.g., dropping one specific booking) are **NOT** permitted.
    *   **Logic:** Cancelling a shift triggers the **Auto-Reallocator** for *all* bookings assigned to that companion during that timeframe.
    *   **Penalty:** Cancelling a shift within a configurable window (e.g., < 4 hours) triggers a **Penalty** on the Companion's account.

*   **1.2.2.4 Pre-Service Dashboard:** Starting **30 minutes prior** to start, companions see:
    *   Client Nickname.
    *   Backup PIN (for manual entry if QR fail).
    *   Countdown Clock showing time remaining until the session starts.

*   **1.2.2.5 Mandatory Geolocation:** GPS must remain enabled at all times as per job role. The app uses this to verify the Duo has reached the assigned mall and reporting location.

*   **1.2.2.6 Duo Readiness (T-30m):** Both the Captain and Companion must arrive at the mall, meet each other ("Team Match"), and signal readiness **30 minutes before** the booking start.
    *   **Failure Protocol ("Breach"):** If readiness is not signaled by T-30m, the specific Duo is marked as "Breach".
    *   **System Action:** An **Admin Request** is raised immediately.
    *   **Resolution:** Admin manually triggers a **"Hot Swap"** to reassign the booking to another available Duo (if available).
    *   **Client Experience:** If swapped successfully, the client sees no change. If no swap is possible, Admin intervenes (Manual cancellation/Refund).

*   **1.2.2.7 Handshake Mechanism (QR Scan):** To officially start the session, the Companion/Captain scans a QR code presented by the Client's app. This verifies the match and "starts the clock."

*   **1.2.2.8 Meeting Fallback:** If the Client denies GPS permissions, the system assigns a specific meeting location within the mall; both the Duo and the Client are navigated to this point.

*   **1.2.2.9 Client Lateness Policy:** If the client arrives late, the 2-hour session clock still begins at the **Scheduled Booking Time**. The session will finish 2 hours after the scheduled start, regardless of arrival time.

#### 1.2.3 System Logic & Matching
*   **1.2.3.1 Matching Params:** Initially "First-Available." Logic is built to support future weighting for Client and Companion ratings.
*   **1.2.3.2 Duo Integrity:** A booking is only valid if both a Captain and a Companion are available for the full duration.

#### 1.2.4 Admin & Logging
*   **1.2.4.1 Manual Controls:** Admin can manually assign, re-assign, update, or cancel any booking.
*   **1.2.4.2 Audit Snapshots:** All bookings log Device ID, Pricing Engine version, and Client location at time of booking.
