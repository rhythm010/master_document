## Scope
- Booking process from the moment user opens the booking page
- Allocation process for companions
- Booking success flows
- Booking cancellation flows / rebooking flows
- Venue types: Malls, Clubs, Restaurants (any partnered location)

### 1.2 The Booking & Allocation Flow
#### 1.2.1 Functional (Client)

##### 1.2.1.1 Entry & Account Verification
*   **1.2.1.1.1 Flow:** Client opens the app and navigates to the "Hire Companion" section.
*   **1.2.1.1.2 Request:** **FE Check:** The system checks the client's current status.
    *   **Logic:** If a client has *any* active or future booking (Status: `PENDING`, `CONFIRMED`, `ACTIVE`), they are **automatically redirected** to the **Booking Details Page** for that booking. They cannot access the "New Booking" flow.
*   **1.2.1.1.3 Logic:** A single client account is strictly serialized to one booking at a time.
*   **1.2.1.1.4 Pending Flow Re-entry:** (Covered by 1.2.1.1.2 - Automatic Redirect).

##### 1.2.1.2 Venue Selection
*   **1.2.1.2.1 Input:** Client types in a venue name. The system fetches matching results from a **pre-defined list** of partnered locations (malls, clubs, restaurants).
*   **1.2.1.2.2 Request:** **[GET] /venues?q={searchTerm}** — autocomplete search against the partnered venues database.
*   **1.2.1.2.3 Geolocation:** System requests GPS; FE sorts results with the nearest venue at the top. **Global Visibility:** All partnered venues are listed regardless of distance/country.
*   **1.2.1.2.4 Venue Data:** Each venue in the list includes: Name, Address, Type (Mall/Club/Restaurant), and general Operating Hours (for informational display only).
*   **1.2.1.2.5 Geo-Fencing:** All selectable venues are pre-defined and geo-fenced by default. No dynamic geo-fence hit-detection is required.
*   **1.2.1.2.6 Logic:** Bookings are specific to a venue and cannot be transferred. Changes require cancellation.
*   **1.2.1.2.7 Venue Responsibility:** The system's role is to supply companions. If a venue is unexpectedly closed (e.g., holiday, private event), it is the **client's responsibility** to verify the venue is open before booking. The system maintains general operating hours for reference but does not guarantee venue availability.

##### 1.2.1.3 Scheduling & Availability
*   **1.2.1.3.1 Input:** Client selects a **Date** and a **Start Time**.
    *   **Booking Window:** 
        *   **Minimum Lead Time:** 24 Hours.
        *   **Maximum Advance:** 14 Days (2 Weeks).
*   **1.2.1.3.2 Request:** **[GET] /availability?venueId={id}&date={date}** — returns available time slots with pricing.
*   **1.2.1.3.3 Source:** Availability is derived from the **General Roster Engine**. The roster is **not location-based** — it tracks the general availability pool of companions. The system intersects roster availability with the venue's general operating hours to produce valid slots.
*   **1.2.1.3.4 Logic:**
    *   **Fixed Duration:** All bookings are for a **Fixed 2-Hour Duration**. Users cannot customize the duration.
    *   **General Roster:** The availability pool is not mapped to specific venues. Any available companion can be assigned to any venue. Travel time and buffer are handled as part of the companion matching flow (see 2.1).
    *   **Pricing Bundled:** The response includes pricing data for each available slot. The **Pricing Engine** fetches the venue type and returns the appropriate pricing. Currently flat-rate; future iterations may vary by venue type, time slot, or demand.
    *   **Anonymity:** No specific Companion identity is exposed or assigned at this stage.
    *   **Static Data:** No auto-refresh logic is applied to this view.
*   **1.2.1.3.5 "Book Now" Visibility:** The "Book Now" button becomes visible only after the client has selected a venue, date, and time slot, and availability + pricing have been confirmed and displayed.

##### 1.2.1.4 Review & Mandatory Consent
*   **1.2.1.4.1 Summary:** Once availability is confirmed and the "Book Now" button is visible, the FE displays:
    *   **Context:** Venue Name, Venue Type, Date, Time.
    *   **Financials:** Full Price Breakdown (Base Rate + VAT + Service Fees = **Grand Total**). Pricing is returned by the availability API alongside the slot data.
    *   **Promo Codes:** Not supported in this version.
*   **1.2.1.4.2 Navigation:**
    *   **Edit:** User must navigate "Back" to modify Date/Time/Venue.
    *   **Timer:** No countdown timer or soft-lock on the slot at this stage.
*   **1.2.1.4.3 Action:** Clicking "Book Now" triggers the **Mandatory Consent Modal**.
*   **1.2.1.4.4 Consent Modal (FE Only):**
    *   **Purpose:** Purely a front-end display to inform the client of behavioural expectations. It has **no relation to booking logic** — no backend call is made for consent.
    *   **Visuals:** A list of "Do Not" items (e.g., No Touch, No Harassment) displayed with illustrative images/icons.
    *   **Mechanism:** User must tick a single checkbox: **"I agree to all terms and conditions"**.
    *   **Enforcement:** The "Confirm & Pay" button is disabled until the checkbox is ticked.

##### 1.2.1.5 Booking Initiation & Payment (Phase 1)
*   **1.2.1.5.1 Request:** **[POST] /bookings** (VenueID, Slot, Nickname).
    *   **Allocation Engine:** The request is sent to the allocation engine for auto-assignment of a companion duo from the available pool.
    *   **Admin API Extension:** The allocation engine also supports optional `captain_id` and/or `vice_captain_id` parameters for selective companion assignment. This is **only available via the Admin API** for future features. From the client FE, both companions are always auto-assigned.
*   **1.2.1.5.2 BE Operations:**
    *   Create record with `status: PENDING`.
    *   **Soft-Lock:** Apply a 15-minute soft-lock on the specific companion pair (Captain + Vice Captain) for the selected time slot. This prevents other clients from booking the same companions for the same slot while the payment is in progress. No other client will see these companions as available during the soft-lock window.
    *   Generate unique Booking ID.
*   **1.2.1.5.3 FE Simulation:** Phase 1 uses **Mock Data** for payments. 
    *   **UI:** Display a "Simulating Payment Authorization" spinner (3-5 seconds).
    *   **Logic:** System mocks a successful "Payment Hold" for the full estimated amount.
*   **1.2.1.5.4 Soft-Lock to Confirmed Transition:**
    *   Once the payment flow completes successfully and the API returns a **200 response**, the soft-lock converts into a confirmed booking.
    *   **Transition:** Booking status changes from `PENDING` → `CONFIRMED`. The `captain_id` and `vice_captain_id` are now permanently assigned.
    *   If the soft-lock expires (15 minutes) without successful payment, the companions are released and the booking status → `FAILED`.
*   **1.2.1.5.5 Pending View Actions:** While in the `PENDING` state (payment in progress), the user is presented with a simplified dashboard. The only available action is a "Cancel Booking" button, which terminates the `PENDING` state and releases the soft-lock.

##### 1.2.1.6 Allocation & Confirmation Sync
*   **1.2.1.6.1 FE Polling:** FE polls **[GET] /bookings/{id}/status** (background auto-retries enabled) or listens for Realtime update.
*   **1.2.1.6.2 Success Path:** 
    *   **Logic:** Allocation engine finalizes the Duo match.
    *   **Transition:** The "Processing" page automatically converts into the **Booking Confirmation Page** without a redirect.
    *   **Notification:** Push notification sent: "Booking Confirmed".
*   **1.2.1.6.3 Failure Path:**
    *   **Logic:** If allocation fails (timeout or race condition), the system **Voids the Mock Transaction**.
    *   **UI Feedback:** The page displays an error message: "Booking cannot be completed. Slot no longer available."
    *   **Retry:** User must restart the flow to select a different slot.
*   **1.2.1.6.4 Companion Duo Assignment:**
    *   **Logic:** Once the client completes the booking, the system assigns the slot to a Companion Duo.
    *   **Roles:** One companion is designated as the **Captain**, and the other as the **Vice Captain**.
    *   **Booking Object:** The booking object for both companions includes:
        *   **Captain Details:** Information about the Captain Companion.
        *   **Vice Captain Details:** Information about the Vice Captain Companion.
        *   **Client Booking Details:** Includes the client ID, booking details, and session ID.
    *   **Pre-Service Matching:** Before the service starts, both companions must match with each other to activate the Companion Duo. This ensures readiness and synchronization for the session.
*   **1.2.1.6.5 Matching Clarification:**
    *   **Client-to-Companions Match** occurs at booking time: the allocation engine assigns a Captain and Vice Captain, transitioning the booking from `PENDING` to `CONFIRMED`.
    *   **Companion-to-Companion Match** occurs at T-30m (day of service): the two assigned companions physically meet and scan QR to activate as a Duo. This is a separate lifecycle tracked via `duo_status`.

##### 1.2.1.7 Final Confirmation & Information Display
*   **Moved to [1.3 Booking Confirmation Page](1.3_Booking_Confirmation_Page.md).** This section is now its own screen-level sub-flow document covering information display, available actions, day-of-service QR/PIN reveal, and page lifecycle.

##### 1.2.1.8 Session Maintenance & Termination
*   **1.2.1.8.1 Cancellation:**
    *   Clients can cancel up to 7 hours before the start time for a **partial refund**. Cancellations within 3 hours of the start time are **non-refundable**.
    *   Re-allocation Flow: If a companion is missing, the system triggers the **Re-allocation Flow** to assign a replacement. The missing companion's record is updated as a **violation**.
*   **1.2.1.8.2 Identity:** The system uses the client's profile identity. The client can set a nickname which will be visible to the companions.
*   **1.2.1.8.3 Admin Logic:** Admin has the authority to cancel a client's booking (triggering a notification). Admin cannot "update" a client booking, only cancel.

##### 1.2.1.9 Accessibility & Localization
*   **1.2.1.9.1 Standards:** Initial implementation follows standard OS accessibility settings; no bespoke luxury-tier accessibility features are planned for Phase 1.
*   **1.2.1.9.2 Localization:** App UI and AI chat agent support **only English and Arabic** as of now.

#### 1.2.2 Functional (Companion) 
*   **1.2.2.1 Calendar & Blocking:** 
    *   **Logic:** Upon booking confirmation, the system **Auto-Assigns** the booking to the Duo. There is **no option to Decline** or Accept.
    *   **Block Duration:** The system blocks the Companion's calendar for **2 Hours (Session)** + **20 Minutes (Rest Buffer)**.
    *   **Notification:** The Companion does **NOT** receive an immediate notification of the booking.
    *   **Visibility:** Booking details (client nickname, timing, meeting location) remain hidden until **X Hours** (Configurable) before the start time. This is done to **protect client privacy**.
    *   **Calendar View:** Before the X-hour reveal window, companions only see a blocked time slot on their calendar with no details — they know they have a booking but cannot see who or where.
    *   **Standard Reporting Time:** Companions have a standard reporting time on the booking day based on their Roster/Shift.
    *   **Travel Time:** Since the roster is general (not location-locked), a **buffer travel time** is provided for companions to reach the assigned venue. Travel time management is part of the companion self-matching flow (see 2.1).
*   **1.2.2.2 Preparation Alerts:** A "Battery Check" push notification is sent a few hours before the booking start time to ensure companions have ample time to reach 100% charge.
*   **1.2.2.3 Shift Context & Cancellation:**
    *   **Context:** Companions operate in **Shifts** (e.g., 6 hours), within which multiple **Bookings** (e.g., 2 hours each) are assigned.
    *   **Cancellation Unit:** Companions can only cancel their **Entire Shift**. Partial cancellations (e.g., dropping one specific booking) are **NOT** permitted.
    *   **Logic:** Cancelling a shift triggers the **Auto-Reallocator** for *all* bookings assigned to that companion during that timeframe.
    *   **Penalty:** Cancelling a shift within a configurable window (e.g., < 4 hours) triggers a **Penalty** on the Companion's account.

*   **1.2.2.4 Pre-Service Dashboard:** Starting **30 minutes prior** to start, companions see:
    *   Client Nickname.
    *   Backup PIN (for manual entry if QR fail).
    *   Countdown Clock showing time remaining until the session starts.

*   **1.2.2.5 Mandatory Geolocation:** GPS must remain enabled at all times as per job role. The app uses this to verify the Duo has reached the assigned venue and reporting location.

*   **1.2.2.6 Duo Readiness (T-30m):** Both the Captain and Vice Captain must arrive at the venue, meet each other ("Team Match"), and signal readiness **30 minutes before** the booking start.
    *   **Failure Protocol ("Breach"):** If the companions have not successfully matched by **T-20m** (20 minutes before booking start), the booking is **automatically cancelled** by the system.
    *   **System Action:** The booking status transitions to `CANCELLED`. The client receives a cancellation notification and a full refund is processed.
    *   **Violation:** Both companions (or the missing companion) receive a violation record.
    *   **Client Impact:** The client is free to make a new booking immediately after cancellation.
    *   **Note:** Hot Swap logic (reassigning to another Duo) is not supported in this version.

*   **1.2.2.7 Handshake Mechanism (QR Scan):** To officially start the session, the Companion/Captain scans a QR code presented by the Client's app. This verifies the match and "starts the clock."

*   **1.2.2.8 Meeting Fallback:** If the Client denies GPS permissions, the meeting point is handled via the client-companion matching page (see 2.2.1). The client can provide manual instructions or an image. Companions can send in-app notifications to the client as a fallback.

*   **1.2.2.9 Client No-Show / Lateness Policy:**
    *   If the client does not arrive and complete the QR handshake with the companions within **15 minutes after the scheduled booking time (T+15m)**, the booking is **automatically marked as completed**.
    *   No refund is issued for a no-show.
    *   Companions are released and free for their next booking, just as they would be after a normal service completion.
    *   The client's `current_booking_id` is cleared and they are free to make a new booking.
    *   If the client arrives late but before T+15m, the session still ends at the original scheduled end time (start + 2 hours). No extension is given.

#### 1.2.3 System Logic & Allocation Engine
*   **1.2.3.1 Allocation Modes:** The allocation engine supports the following modes:
    *   **Mode 1 — Full Auto (Client Default):** No companion IDs provided. System selects both Captain and Vice Captain from the available pool. This is the only mode available from the client FE.
    *   **Mode 2 — Specific Captain (Admin Only):** `captain_id` is provided in the request. System selects the Vice Captain freely from the available pool.
    *   **Mode 3 — Specific Vice Captain (Admin Only):** `vice_captain_id` is provided in the request. System selects the Captain freely from the available pool.
    *   **Mode 4 — Both Specified (Admin Only):** Both `captain_id` and `vice_captain_id` are provided. System validates both are available and assigns directly.
*   **1.2.3.2 Matching Params:** Initially "First-Available" for auto modes. Logic is built to support future weighting for Client and Companion ratings.
*   **1.2.3.3 Duo Integrity:** A booking is only valid if both a Captain and a Vice Captain are available for the full duration.
*   **1.2.3.4 General Roster:** The roster is **not location-based**. Companions are available in a general pool. The system ensures travel time + buffer is factored in during the pre-deployment phase (see 2.1).
*   **1.2.3.5 Matching Flow:** The detailed companion-to-companion matching flow is covered in **[2.1 Companion Self-Matching](2.1_Companion_Self_Matching.md)**.

#### 1.2.4 Admin & Logging
*   **1.2.4.1 Manual Controls:** Admin can manually assign, re-assign, or cancel any booking. Re-assignment uses the allocation engine's selective modes (Modes 2–4) to replace one or both companions on a confirmed booking.
*   **1.2.4.2 Re-Assignment Use Case:** When a companion cannot make it (e.g., emergency), admin can use selective allocation to assign a replacement via the API without cancelling the booking.
*   **1.2.4.2 Audit Snapshots:** All bookings log Device ID, Pricing Engine version, and Client location at time of booking.
*   **1.2.4.3 Edge Cases:**
    *   **Client No-Show:** If the client does not show up and complete the QR handshake within **T+15m** (15 minutes after scheduled start), the booking is automatically marked as completed, no refund is issued, and companions are released.
    *   **Client GPS Denial:** This edge case will be handled in a future update.
