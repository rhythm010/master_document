## 1. System Overview

### 1.1 Summary of Backend Responsibilities

The backend serves as the API layer for the Project Companion booking system. It handles:

- Client authentication via OTP
- Venue search and listing
- Availability calculation from the general companion roster
- Booking creation with companion allocation (auto and admin-selective)
- Payment simulation (Phase 1 mock)
- Soft-lock lifecycle management (15-minute window)
- Booking status transitions (`PENDING` → `CONFIRMED` → `ACTIVE` → `COMPLETED`)
- Cancellation with tiered refund calculation
- Companion calendar blocking (2h session + 20m buffer)
- Scheduled jobs: soft-lock expiry, duo breach auto-cancel (T-20m), client no-show auto-complete (T+15m)
- Push notification dispatch
- Audit logging for every booking action
- Admin endpoints for manual assignment, reassignment, and cancellation

### 1.2 In Scope

|------|---------|
| Venue search API | Autocomplete search against partnered venues |
| Availability API | Slot computation from general roster, pricing bundled |
| Booking CRUD | Create, read, cancel bookings |
| Allocation engine | 4 modes (auto, captain-specified, vice-captain-specified, both-specified) |
| Soft-lock lifecycle | 15-minute lock on companion pair during payment |
| Payment mock | Simulated hold/charge/void/refund (Phase 1) |

## 2. High-Level Architecture

### 2.1 Architecture Style

**Monolithic REST API.** A single Node.js/TypeScript process serving all endpoints.

Justification:
- Zero users exist today. A monolith eliminates deployment, networking, and service-discovery complexity.
- A single codebase is easier for AI agents to navigate and modify.
- All domain logic (booking, allocation, venues) shares the same database — no cross-service calls needed.
- Refactoring into services later is straightforward because the internal module boundaries are clean.

| Component | Responsibility |
|-----------|---------------|
| **Auth Module** | OTP generation, verification, JWT issuance, token refresh |
| **Venue Module** | Venue search, listing, detail retrieval |
| **Booking Module** | Booking CRUD, status transitions, cancellation, refund calculation |
| **Admin Module** | Admin-only endpoints for assignment, reassignment, cancellation |
| **Allocation Engine** | Companion selection logic (4 modes), availability intersection |
| **Pricing Engine** | Flat-rate pricing by venue type (extensible) |
| **Refund Engine** | Tiered refund calculation based on time-to-start |
| **Soft-Lock Manager** | 15-minute lock lifecycle, expiry handling |
| **Scheduler** | Cron jobs for automated state transitions |
| **Notification Service** | Interface for push notifications (actual delivery abstracted) |
| **Prisma ORM** | Database access, migrations, type-safe queries |

### 2.3 Why This Is Intentionally Simple

- One deployable artifact.
- One database.
- No message queues, no event buses, no service mesh.
- Cron jobs run in-process (no external scheduler).
- Push notification dispatch is a function call to an interface — swap the implementation later.
- No caching layer. PostgreSQL handles the query load for <1000 users trivially.

### Justifications

**Express.js over Fastify/Hono/Nest:**
- Most widely adopted Node.js framework. Every developer and AI agent has deep familiarity.
- Enormous middleware ecosystem. Every problem has a solved Express middleware.
- Debugging is straightforward — middleware is a simple function chain.
- No decorator magic, no DI containers, no abstraction layers to trace through.

**Prisma over Drizzle/TypeORM/Knex:**
- TypeScript-first with auto-generated types from schema. Zero type drift between DB and code.
- Declarative schema file is readable by both humans and AI agents.
- Built-in migration system. No separate migration tool needed.
- Prisma Studio provides a free GUI for data inspection during development.
- Widely adopted (30k+ GitHub stars), well-documented.

**PostgreSQL over MySQL/SQLite:**
- Robust ACID transactions — critical for soft-lock and concurrent booking scenarios.
- Native UUID type, JSON/JSONB columns, and advanced indexing.
- Excellent tooling (pgAdmin, psql, Prisma Studio).
- Free, open source, production-grade at any scale.

**Zod over Joi/Yup/AJV:**
- TypeScript-native. Schema definition produces TypeScript types directly — no parallel type definitions.
- Composable — schemas can be reused and extended.
- Small bundle, zero dependencies.
- `.parse()` throws; `.safeParse()` returns result. Both patterns available.

**Pino over Winston/Bunyan:**
- Fastest Node.js logger (low overhead in benchmarks).
- JSON-structured output by default — easy to search and parse.
- pino-pretty for human-readable dev logs.
- No transports to configure for basic use.

**Vitest over Jest:**
- Native TypeScript and ESM support without configuration.
- Compatible with Jest API — familiar assertion syntax.
- Faster execution due to Vite-based transform pipeline.
- Built-in code coverage.

**node-cron over Bull/BullMQ/Agenda:**
- In-process scheduling. No Redis dependency. No external workers.
- Cron syntax is universally understood.
- Sufficient for <1000 users with a handful of scheduled checks.
- When the system outgrows in-process cron, migrating to BullMQ is a contained refactor.

| `src/modules/` | Feature-grouped routes, controllers, services, and schemas. Each module is self-contained. Controllers parse/validate, services contain business logic, routes wire them together. |
| `src/engines/` | Pure business logic with no HTTP awareness. Allocation, pricing, and refund are deterministic functions that receive data and return results. |
| `src/schedulers/` | Each file is one cron job. `index.ts` registers all jobs. Jobs call services directly. |
| `src/services/` | External service interfaces. Each has a clear interface and a stub/mock implementation for Phase 1. Swap implementations without changing callers. |
| `src/shared/` | Cross-module utilities. Kept minimal — only genuinely shared code lives here. |
| `src/test/` | Test infrastructure. Tests themselves are co-located with modules (e.g., `booking.service.test.ts` next to `booking.service.ts`). |

### 5.4 Key Constraints

- `clients.current_booking_id` is `UNIQUE` — enforces one active booking per client at the database level.
- `bookings.captain_id` and `bookings.vice_captain_id` are nullable — they are null during the brief window between booking creation and allocation engine response, which happens within the same transaction.
- `booking_audit_logs` has no UPDATE or DELETE operations — append-only by application convention. No DB-level trigger needed for Phase 1; enforce in code.
- All monetary fields use `Decimal(10, 2)` — sufficient for AED amounts up to 99,999,999.99.
- All GPS fields use `Decimal(10, 7)` — accurate to ~1.1cm.
- **Monetary math convention:** All monetary calculations in the application layer (pricing, refund, VAT) must use `Prisma.Decimal` (which wraps `decimal.js`). Never convert monetary Decimals to JavaScript `number` for arithmetic. API responses serialize monetary values as strings (e.g., `"575.00"`).

### 6.2 Venue Endpoints

#### 6.2.1 `GET /api/v1/venues`

Search and list partnered venues.

| Field | Value |
|-------|-------|
| Auth | Client JWT required |

**Query Parameters:**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `q` | string | No | Search term for autocomplete (min 2 chars). Matches against venue name. |
| `latitude` | number | No | Client's latitude for distance sorting |
| `longitude` | number | No | Client's longitude for distance sorting |

**Response 200:**
```json
{
  "success": true,
  "data": {
    "venues": [
      {
        "id": "uuid",
        "name": "Dubai Mall",
        "type": "MALL",
        "address": "Financial Centre Rd, Downtown Dubai",
        "latitude": 25.1972,
        "longitude": 55.2796,
        "country": "AE",
        "operatingHoursStart": "10:00",
        "operatingHoursEnd": "00:00",
        "distanceKm": 2.3
      }
    ]
  }
}
```

**Backend Logic:**
1. Query `venues` where `isActive: true`.
2. If `q` is provided, filter by `name ILIKE '%{q}%'`. Sanitize `q` — Prisma parameterizes automatically.
3. If `latitude` and `longitude` are provided, compute distance using the Haversine formula in a raw SQL expression and sort ascending. Otherwise sort alphabetically.
4. `distanceKm` is `null` if client coordinates not provided.
5. Return all matching venues — no pagination needed for Phase 1 (partnered venue count < 100).

**Haversine SQL (Prisma raw query):**
```sql
SELECT *,
  (6371 * acos(
    cos(radians($1)) * cos(radians(latitude)) *
    cos(radians(longitude) - radians($2)) +
    sin(radians($1)) * sin(radians(latitude))
  )) AS distance_km
FROM venues
WHERE is_active = true
  AND (name ILIKE $3 OR $3 IS NULL)
ORDER BY distance_km ASC NULLS LAST;
```

```json
    "date": "2026-03-01",

        "startTime": "10:00",
        "endTime": "12:00",
        "available": true,
        "pricing": {
          "baseRate": 500.00,
          "vatAmount": 25.00,
          "serviceFee": 50.00,
          "grandTotal": 575.00,
          "currency": "AED"
        }
      },
      {
        "startTime": "12:00",
        "endTime": "14:00",
        "available": true,
        "pricing": {
          "baseRate": 500.00,
          "vatAmount": 25.00,
          "serviceFee": 50.00,
          "grandTotal": 575.00,
          "currency": "AED"
        }
      },
      {
        "startTime": "14:00",
        "endTime": "16:00",
        "available": false,
        "pricing": null
      }
    ]
  }
}
```

**Error 400:**
```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Date must be between 1 and 14 days from now.",
    "details": [{ "field": "date", "message": "Must be at least 24 hours in the future" }]
  }
}
```

**Backend Logic:**

1. Validate venue exists and is active. Return 404 if not.
2. Fetch venue operating hours.
3. Generate 2-hour slot windows within operating hours. A slot at `22:00` requires operating hours to cover until `00:00`.
4. For each slot, determine how many companion **pairs** (duos) are available:
   a. Query all `shifts` for the given `date` where `status = SCHEDULED` or `status = ACTIVE`.
   b. For each shift, check if the companion's availability covers the full slot window (start_time to end_time) **plus** a 30-minute buffer before and 20-minute rest buffer after.
   c. Exclude companions who already have a booking (status `PENDING`, `CONFIRMED`, or `ACTIVE`) overlapping with the slot window + 30m buffer.
   d. Exclude companions who are soft-locked for the slot (booking status `PENDING` with `soft_lock_expires_at > now()`).
   e. Group remaining available companions by role: Captains and Vice Captains.
   f. A slot is `available: true` if at least 1 Captain AND 1 Vice Captain are both free.
5. For available slots, call `pricing.engine.calculatePrice(venueType)` to produce pricing.
6. Return all slots (both available and unavailable) so the FE can display the full schedule.

**Pricing Engine (Phase 1 — Flat Rate):**

| Venue Type | Base Rate (AED) | VAT (5%) | Service Fee (AED) | Grand Total |
|------------|----------------|----------|-------------------|-------------|
| MALL | 500.00 | 25.00 | 50.00 | 575.00 |
| CLUB | 600.00 | 30.00 | 60.00 | 690.00 |
| RESTAURANT | 400.00 | 20.00 | 40.00 | 460.00 |

These values are stored in `src/config/constants.ts`, not in the database. Change requires redeployment. Future iterations may move to DB-configurable pricing.

**Error 409 (client already has active booking):**
```json
{
  "success": false,
  "error": {
    "code": "ACTIVE_BOOKING_EXISTS",
    "message": "You already have an active booking. Cancel it before creating a new one.",
    "data": {
      "currentBookingId": "uuid",
      "currentBookingStatus": "CONFIRMED"
    }
  }
}
```

**Error 409 (slot no longer available):**
```json
{
  "success": false,
  "error": {
    "code": "SLOT_UNAVAILABLE",
    "message": "No companion duo is available for this slot."
  }
}
```

**Backend Logic (within a single database transaction):**

```
BEGIN TRANSACTION (Serializable isolation for critical section)

1. Check client.bookingStatusCache != NONE.
   → If not NONE, return 409 ACTIVE_BOOKING_EXISTS.

2. Re-validate slot availability (same logic as GET /availability, 
   but inside the transaction to prevent race conditions).
   → If no duo available, return 409 SLOT_UNAVAILABLE.

3. Run Allocation Engine (Mode 1: AUTO):
   a. Select one available Captain — "First Available" strategy 
      (order by total_sessions ASC to distribute load).
   b. Select one available Vice Captain — same strategy.
   c. Record their shift IDs for the booking.

4. Calculate pricing via Pricing Engine.

5. Create Booking record:
   - status: PENDING
   - captain_id: selected captain
   - vice_captain_id: selected vice captain
   - soft_lock_expires_at: NOW() + 15 minutes
   - payment_hold_status: NONE
   - client_nickname_snapshot: client.nickname
   - duo_status: PENDING
   - Generate duo_qr_code (UUID-based unique string)
   - Generate duo_pin_code (random 6-digit)
   - Generate qr_code (UUID-based unique string for client-companion matching)
   - Generate pin_code (random 4-digit for client-companion matching)
   - All pricing fields filled

6. Update Client:
   - current_booking_id: new booking ID
   - booking_status_cache: PENDING

7. Create BookingAuditLog:
   - action: CREATED
   - performed_by_type: CLIENT
   - performed_by_id: client.id
   - device_id: from request header
   - pricing_engine_version: "1.0"
   - allocation_engine_version: "1.0"
   - client_latitude/longitude: from request header (if provided)

COMMIT TRANSACTION
```

8. After transaction commits, trigger mock payment flow:
   - Call `payment.service.holdAmount(booking.grandTotal)` — Phase 1 stub always returns success after 3–5 second simulated delay.
   - On success: Call `confirmBooking(bookingId)` (see below).
   - On failure: Call `failBooking(bookingId, "Payment hold failed")`.

**`confirmBooking(bookingId)` logic:**
```
BEGIN TRANSACTION
1. Fetch booking. Verify status == PENDING and soft_lock_expires_at > NOW().
   → If expired, void payment, set status FAILED, return.
2. Update booking:
   - status: CONFIRMED
   - payment_hold_status: HELD
   - payment_hold_amount: grand_total
3. Update client:
   - booking_status_cache: CONFIRMED
4. Create BookingAuditLog: action CONFIRMED.
COMMIT

5. Dispatch notification to client: BOOKING_CONFIRMED.
6. Block companion calendars (captain + vice captain) for date/slot + 20m buffer.
   (Calendar blocking = creating entries that the availability check will see.)
```

**`failBooking(bookingId, reason)` logic:**
```
BEGIN TRANSACTION
1. Update booking:
   - status: FAILED
   - failure_reason: reason
   - payment_hold_status: VOIDED
2. Update client:
   - current_booking_id: null
   - booking_status_cache: NONE
3. Create BookingAuditLog: action FAILED.
COMMIT
```

**Race Condition Handling:**
- The transaction uses `Serializable` isolation level on the critical section (step 2–6). Two simultaneous `POST /bookings` requests for the same slot will serialize: the second will fail availability check after the first has locked the companions.
- Prisma supports `$transaction` with isolation level control:
  ```typescript
  prisma.$transaction(async (tx) => { ... }, { isolationLevel: 'Serializable' })
  ```

**Serialization Failure Retry Strategy:**
- PostgreSQL Serializable isolation may throw `40001` errors (serialization failure) when concurrent transactions conflict. This is expected behavior.
- The backend wraps booking creation in a `withSerializableRetry()` utility:
  ```typescript
  async function withSerializableRetry<T>(
    fn: () => Promise<T>,
    maxRetries = 3,
    baseDelayMs = 100
  ): Promise<T> {
    for (let attempt = 1; attempt <= maxRetries; attempt++) {
      try {
        return await fn();
      } catch (error) {
        if (error.code === '40001' && attempt < maxRetries) {
          await sleep(baseDelayMs * attempt); // 100ms, 200ms, 300ms
          continue;
        }
        throw error;
      }
    }
  }
  ```
- On all retries exhausted, throw `SLOT_UNAVAILABLE`. The logic is idempotent — each retry re-checks availability.

**Idempotency-Key Header:**
- `POST /bookings` accepts an optional `Idempotency-Key` header (UUID).
- Before starting the transaction, check an in-memory `Map<string, { bookingId, response, expiresAt }>` for the key.
- If found: return the cached 201 response. No DB operation.
- If not found: proceed with creation. On success, cache `key → response` with 15-minute TTL.
- If header is absent: proceed without idempotency (backward compatible).
- Phase 2: Migrate idempotency store to Redis (same pattern as OTP store).

```json
  "success": true,
  "data": {
    "bookingId": "uuid",
    "status": "CANCELLED",
    "refundPercentage": 100,
    "refundAmount": 575.00,
    "message": "Booking cancelled. Full refund will be processed."
  }
}
```

**Error 400:**
```json
{
  "success": false,
  "error": {
    "code": "BOOKING_NOT_CANCELLABLE",
    "message": "This booking cannot be cancelled in its current state."
  }
}
```

**Backend Logic:**

```
BEGIN TRANSACTION
1. Fetch booking. Verify ownership.
2. Verify booking.status is PENDING or CONFIRMED. 
   → ACTIVE, COMPLETED, CANCELLED, FAILED → return 400.

3. Calculate refund using Refund Engine:
   hoursUntilStart = (booking.date + booking.startTime) - NOW() in hours

   | Condition              | Refund % |
   |------------------------|----------|
   | hoursUntilStart > 24   | 100%     |
   | 7 <= hoursUntilStart <= 24 | 50% |
   | hoursUntilStart < 7    | 0%       |

   refundAmount = grandTotal * refundPercentage / 100

4. Update booking:
   - status: CANCELLED
   - cancelled_by: CLIENT
   - cancelled_at: NOW()
   - cancellation_reason: request.reason
   - refund_percentage: calculated
   - refund_amount: calculated
   - payment_hold_status: if refundPercentage == 100 → VOIDED; else → REFUNDED

5. Update client:
   - current_booking_id: null
   - booking_status_cache: NONE

6. Create BookingAuditLog: action CANCELLED.
COMMIT

7. Call payment.service.refund(booking.id, refundAmount) — mock in Phase 1.
8. Dispatch notification to client: CANCELLATION.
9. Release companion soft-lock / calendar blocks.
```

#### 6.4.5 `GET /api/v1/bookings/current`

Get client's current active booking (convenience endpoint for redirect logic).

| Field | Value |
|-------|-------|
| Auth | Client JWT required |

```json
{

  "data": {
    "hasActiveBooking": true,
    "bookingId": "uuid",
    "status": "CONFIRMED"
```

**Response 200 (no booking):**

```json
{
  "success": true,

    "bookingId": null,

  }
```

**Backend Logic:**

2. **Reconciliation safety net:** If `currentBookingId` is non-null, fetch the booking's actual status. If the booking is in a terminal state (`COMPLETED`, `CANCELLED`, `FAILED`) but `bookingStatusCache` is not `NONE`, auto-correct the client record (`currentBookingId = null`, `bookingStatusCache = NONE`) within the same request. Log this as a `warn` event — it indicates a state drift that should be investigated.

#### 6.5.1 `POST /api/v1/admin/bookings`

Admin creates a booking with selective companion assignment.

| Field | Value |
|-------|-------|
| Auth | Admin JWT required |

**Request Body:**
```json
{
  "clientId": "uuid",
  "venueId": "uuid",
  "date": "2026-03-01",
  "startTime": "10:00",
  "captainId": "uuid",
  "viceCaptainId": null
}
```

**Validation:**
- `clientId`: Required. Must reference an existing client.
- `venueId`, `date`, `startTime`: Same rules as client endpoint.
- `captainId`: Optional. UUID. Must reference an active, available Captain.
- `viceCaptainId`: Optional. UUID. Must reference an active, available Vice Captain.
- At least one of `captainId` or `viceCaptainId` can be null (system auto-fills). Both can be null (full auto). Both can be provided (full manual).

**Backend Logic:**
1. Determine allocation mode:
   - Both null → `AUTO`
   - Only captain → `CAPTAIN_SPECIFIED`
   - Only vice captain → `VICE_CAPTAIN_SPECIFIED`
   - Both provided → `BOTH_SPECIFIED`
2. Validate specified companions are available for the slot.
3. Run allocation engine with the determined mode. Auto-fill any unspecified role.
4. Create booking with `status: CONFIRMED` directly (admin bypasses payment).
5. Set `payment_hold_status: HELD`, `payment_hold_amount: grand_total`.
6. Update client's `current_booking_id` and `booking_status_cache`.
7. Audit log with `performed_by_type: ADMIN`.

**Response 201:** Same shape as client booking creation response, but `status: CONFIRMED`.

## 7. Authentication & Authorization Strategy

### 7.1 Token Design

| Property | Client Token | Admin Token | Companion Token |
|----------|-------------|-------------|-----------------|
| Algorithm | HS256 | HS256 | HS256 |
| Expiry | 1 hour | 8 hours | 8 hours |
| Refresh | 30-day refresh token | No refresh (re-login) | No refresh (re-login) |
| Payload | `{ sub: clientId, role: "CLIENT" }` | `{ sub: adminId, role: "SUPER_ADMIN" \| "SUPPORT" \| "OPERATIONS" }` | `{ sub: companionId, role: "COMPANION" }` |
| Secret | `JWT_SECRET` env var | Same secret | Same secret |

All three token types share the same signing secret for simplicity. The `role` field in the payload distinguishes them.

### 7.2 Middleware Design

```typescript
// authenticate.ts — Verifies JWT, attaches user to request
// Applied to all routes except /auth/*

function authenticate(req, res, next) {
  const token = req.headers.authorization?.split("Bearer ")[1];
  if (!token) return res.status(401).json({ success: false, error: { code: "NO_TOKEN" } });
  
  try {
    const payload = jwt.verify(token, JWT_SECRET);
    req.user = { id: payload.sub, role: payload.role };
    next();
  } catch {
    return res.status(401).json({ success: false, error: { code: "INVALID_TOKEN" } });
  }
}

// authorize.ts — Checks role against allowed roles
// Usage: router.post("/admin/bookings", authorize("SUPER_ADMIN", "OPERATIONS"), controller)

function authorize(...allowedRoles: string[]) {
  return (req, res, next) => {
    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({ success: false, error: { code: "FORBIDDEN" } });
    }
    next();
  };
}
```

### 7.3 Route Protection Matrix

| Route Pattern | Allowed Roles |
|--------------|---------------|
| `POST /auth/*` | None (public) |
| `GET /venues` | CLIENT |
| `GET /availability` | CLIENT |
| `POST /bookings` | CLIENT |
| `GET /bookings/:id/*` | CLIENT (ownership verified) |
| `POST /bookings/:id/cancel` | CLIENT (ownership verified) |
| `GET /bookings/current` | CLIENT |
| `GET /companion/bookings*` | COMPANION |
| `POST /admin/*` | SUPER_ADMIN, OPERATIONS |

### 7.4 OTP Storage (Phase 1)

Phase 1 uses an in-memory `Map<string, { hash: string, expiresAt: number }>` for OTP storage.

- Key: phone number.
- Value: bcrypt hash of the OTP + expiry timestamp.
- Entry is deleted after successful verification or expiry.
- In Phase 2, migrate to Redis with TTL.

### 8.2 Global Error Handler

```typescript
// src/middleware/error-handler.ts
// Must be registered LAST in Express middleware chain.

function errorHandler(err: Error, req: Request, res: Response, next: NextFunction) {
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({
      success: false,
      error: {
        code: err.code,
        message: err.message,
        details: err.details ?? undefined,
      },
    });
  }

  // Prisma known errors
  if (err.code === "P2002") { // Unique constraint violation
    return res.status(409).json({
      success: false,
      error: { code: "DUPLICATE_ENTRY", message: "Resource already exists" },
    });
  }

  // Unexpected errors — log full stack, return generic message
  logger.error({ err, requestId: req.id }, "Unhandled error");
  return res.status(500).json({
    success: false,
    error: { code: "INTERNAL_ERROR", message: "An unexpected error occurred" },
  });
}
```

### 8.3 Validation Middleware

```typescript
// src/middleware/validate.ts

function validate(schema: ZodSchema) {
  return (req: Request, res: Response, next: NextFunction) => {
    const result = schema.safeParse({
      body: req.body,
      query: req.query,
      params: req.params,
    });

    if (!result.success) {
      const details = result.error.issues.map((issue) => ({
        field: issue.path.join("."),
        message: issue.message,
      }));
      throw new ValidationError(details);
    }

    req.validated = result.data;
    next();
  };
}
```

### 9.1 Logger Setup

```typescript

  level: process.env.LOG_LEVEL || "info",

      : undefined, // JSON in production
});
```

### 9.2 What Gets Logged

| Level | What | Example |
|-------|------|---------|
| `error` | Unhandled exceptions, failed DB transactions, payment failures | `"Booking confirmation failed: soft-lock expired"` |
| `warn` | Soft-lock expirations, failed OTP attempts, approaching rate limits | `"Soft-lock expired for booking abc-123"` |
| `info` | Booking created, confirmed, cancelled; cron job runs; API requests | `"Booking abc-123 confirmed for client xyz"` |
| `debug` | Allocation engine decisions, availability slot computation, query timing | `"Allocation: selected captain C1, vice captain C2 from pool of 5"` |

### 9.3 HTTP Request Logging

Every request/response is logged via `pino-http` middleware:
- Request: method, URL, request ID (generated UUID), user ID (from JWT if present).
- Response: status code, response time in ms.

### 9.4 What NOT to Log

- OTP values (neither plaintext nor hash).
- JWT tokens.
- Payment method tokens.
- Full request bodies with sensitive fields (phone numbers are allowed; card data would not be).
- GPS coordinates in production logs (store in tracking tables, not in log files).

### 9.5 Request ID

Every request gets a UUID via a middleware. This ID is:
- Attached to the Pino logger context for the request lifecycle.
- Returned in the `X-Request-Id` response header.
- Passed to all audit log entries created during the request.

|-----------|---------------|-------------|
| In-memory OTP store | Multi-instance deployment | Migrate to Redis with TTL keys |
| In-process cron jobs | Multi-instance (duplicate job execution) | Migrate to BullMQ with Redis backend |
| Serializable transactions | >50 concurrent booking requests | Add SELECT ... FOR UPDATE row-level locking, or application-level distributed locks via Redis |
| Flat-rate pricing | Business wants dynamic pricing | Move pricing config to database table, add time/demand multipliers |
| Single DB instance | >10,000 active bookings, >100 queries/sec | Add read replicas (PG streaming replication) |
| No caching | Repeated venue searches under high traffic | Add Redis cache for venue list (5-min TTL) |
| Mock payment | Real money needed | Integrate Stripe/PayTabs with the same `payment.service` interface |
| Prisma query performance | Complex availability queries slow down | Add materialized views or denormalized availability table |

### 13.2 How to Upgrade Without Changing Core Architecture

- **Add Redis:** Install `ioredis`. Replace `Map` in OTP store. Replace `node-cron` with `bullmq`.
- **Add real payments:** Implement `payment.service.ts` interface with Stripe/PayTabs SDK.
- **Add real SMS:** Implement `sms.service.ts` interface with Twilio/MessageBird SDK.
- **Add real push notifications:** Implement `notification.service.ts` with Firebase Admin SDK.
- **Add WebSocket for real-time status:** Install `socket.io`. Emit events from booking service when status changes. FE listens instead of polling.
- **Add rate limiting:** Install `express-rate-limit` with Redis store.

All of these are **additive changes** — no existing code needs restructuring.

## 15. Scheduled Jobs Summary

| Job | Cron Expression | Logic |
|-----|----------------|-------|
| **Soft-Lock Expiry** | `* * * * *` (every minute) | Find bookings where `status = PENDING` AND `soft_lock_expires_at < NOW()`. For each: set `status = FAILED`, `failure_reason = "Soft-lock expired"`, void mock payment, reset client's `current_booking_id` and `booking_status_cache`, audit log. |
| **Duo Breach Auto-Cancel** | `* * * * *` (every minute) | Find bookings where `status = CONFIRMED` AND `duo_status != ACTIVATED` AND `(date + start_time - 20 minutes) < NOW()`. For each: set `status = CANCELLED`, `cancelled_by = SYSTEM`, `duo_status = BREACH`, issue 100% refund, reset client, create violation records for both companions, audit log. |
| **Client No-Show Auto-Complete** | `* * * * *` (every minute) | Find bookings where `status = CONFIRMED` AND `duo_status = ACTIVATED` AND `session_started_at IS NULL` AND `(date + start_time + 15 minutes) < NOW()`. For each: set `status = COMPLETED`, `client_no_show = true`, `session_ended_at = NOW()`, charge payment (mock), reset client, increment companion `total_sessions`, audit log. |
| **Booking Detail Reveal** | `*/5 * * * *` (every 5 min) | Find bookings where `status = CONFIRMED` AND detail not yet revealed AND `(date + start_time - COMPANION_DETAIL_REVEAL_HOURS) < NOW()`. Mark as revealed (internal flag or inferred from time check on read). |
| **Battery Check Notification** | `0 * * * *` (every hour) | Find companions with shifts starting in ~4 hours. Send `BATTERY_CHECK` notification. |
| **Location Log Cleanup** | `0 3 * * *` (daily at 3 AM) | Delete `LocationTrackingLog` records where `created_at < NOW() - 90 days`. Retention policy: 90 days is sufficient for dispute resolution. |

## 17. Allocation Engine Detail

### 17.1 Interface

```typescript
interface AllocationRequest {
  venueId: string;
  date: Date;
  startTime: string;    // HH:mm
  endTime: string;      // HH:mm
  captainId?: string;   // Mode 2 or 4
  viceCaptainId?: string; // Mode 3 or 4
}

interface AllocationResult {
  captainId: string;
  viceCaptainId: string;
  captainShiftId: string;
  viceCaptainShiftId: string;
  mode: AllocationMode;
}

function allocate(
  request: AllocationRequest,
  tx: PrismaTransactionClient
): Promise<AllocationResult>;
```

### 17.2 Algorithm (Mode 1: AUTO)

```
1. Query all shifts for the given date where status IN (SCHEDULED, ACTIVE).
2. For each shift, check if the companion meets ALL criteria:
   a. is_active = true
   b. background_verified = true
   c. penalty_status != PENALIZED
   d. Shift covers: (slot.startTime - INTER_BOOKING_BUFFER) to (slot.endTime + REST_BUFFER)
   e. No existing booking in status (PENDING, CONFIRMED, ACTIVE) that overlaps 
      with the window: (slot.startTime - INTER_BOOKING_BUFFER) to (slot.endTime + REST_BUFFER)
      
3. Split available companions into two lists: Captains and Vice Captains.

4. If either list is empty → throw SLOT_UNAVAILABLE.

5. Sort each list by total_sessions ASC (distribute workload evenly).

6. Select first Captain and first Vice Captain.

7. Return allocation result with their IDs and shift IDs.
```

### 17.3 Algorithm (Mode 2: CAPTAIN_SPECIFIED)

```
1. Validate specified captain_id meets all criteria from step 2 above.
   → If not: throw COMPANION_UNAVAILABLE.
2. Select Vice Captain using Mode 1 logic (step 3-6) from available pool.
3. Return.
```

Modes 3 and 4 follow the same pattern.

### 17.4 Overlap Detection

Two time windows overlap if:
```
existing.start < new.end AND existing.end > new.start
```

Applied to the expanded booking window (start - buffer, end + rest buffer).
