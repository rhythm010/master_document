## Scope
- End-to-end happy flow: from selecting a location and time slot to receiving the booking confirmation message
- Allocation of suitable companions on the basis of availability once the booking request is created
- Major focus on the happy flow for now

### 1.2 The Booking & Allocation Flow
#### 1.2.1 Functional (Client)

##### 1.2.1.1 Location & Time Slot Selection
*   **Pre-condition:** Client has no active or future booking (`CONFIRMED`, `ACTIVE`). One booking at a time per account.

**A. Location Selection**
*   **1.2.1.1.1 Input:** Client types a venue name. The system returns matching results from a **pre-defined list** of partnered locations (malls, clubs, restaurants).
*   **1.2.1.1.2 Request:** **[GET] /venues?q={searchTerm}** — autocomplete search against the partnered venues database.
*   **1.2.1.1.3 Venue Data:** Each result includes: Name, Address, Type (Mall/Club/Restaurant), and general Operating Hours.
*   **1.2.1.1.4 Lock-in:** Bookings are tied to a venue. Changes require cancellation and rebooking.

**B. Time Slot Selection**
*   **1.2.1.1.5 Input:** Client selects a **Date**. The system fetches and displays available start times for the selected venue and date.
*   **1.2.1.1.6 Request:** **[GET] /availability?venueId={id}&date={date}** — returns available start-time slots based on companion availability at the selected location.
*   **1.2.1.1.7 Availability Page:** The client sees a list of available start times. A slot qualifies as available if a companion duo is available at that venue for the next 2 hours from that start time. Availability is derived from the **venue's roster**, intersected with the venue's operating hours.
*   **1.2.1.1.8 Selection:** Client picks a start time from the available options.
*   **1.2.1.1.9 Key Rules:**
    *   **Fixed Duration:** All bookings are **2 hours**. The client only selects a start time — the UI displays that a minimum 2-hour session is booked.
    *   **Pricing:** Hardcoded static amount displayed on the page.
    *   **Anonymity:** No companion identity is exposed at this stage.

##### 1.2.1.2 Book Now & Consent
*   **1.2.1.2.1 Book Now Page:** Once the client has selected a venue, date, and start time, a **"Book Now" page** is displayed showing:
    *   Selected Venue, Date, and Time.
    *   Hardcoded price for the session.
*   **1.2.1.2.2 Action:** Client taps the **"Book Now"** button on this page, which triggers the **Consent Modal**.
*   **1.2.1.2.3 Consent Modal (FE Only):**
    *   Hardcoded UI modal displaying behavioural expectations (e.g., No Touch, No Harassment) — no backend call involved.
    *   Client must tick a single checkbox: **"I agree to all terms and conditions"**.
    *   **"Confirm Booking"** button is disabled until the checkbox is ticked.
*   **1.2.1.2.4 Next Step:** On tapping "Confirm Booking", the booking request is created (see 1.2.1.3).

##### 1.2.1.3 Booking Request & Confirmation
*   **1.2.1.3.1 Request:** **[POST] /bookings** (VenueID, Slot).
    *   The request is sent to the **Allocation Engine**, which checks the venue roster and auto-assigns a companion duo (Captain + Vice Captain) from the available pool.
    *   **Admin Note:** The allocation engine also supports optional `captain_id` / `vice_captain_id` for selective assignment via the Admin API. From the client FE, both companions are always auto-assigned.
*   **1.2.1.3.2 BE Operations (Sync):**
    *   Allocation engine checks the **venue roster** for availability at the selected time slot.
    *   Assigns a Captain and Vice Captain.
    *   Creates booking record with `status: CONFIRMED`.
    *   Generates a unique Booking ID.
    *   Returns **Booking ID + CONFIRMED status** to the FE.
*   **1.2.1.3.3 FE Redirect:** On receiving the confirmed response, the client is redirected to the **[Booking Confirmation Page (1.3)](1.3_Booking_Confirmation_Page.md)**.
*   **1.2.1.3.4 Notification:** Push notification sent to the client: "Booking Confirmed".
*   **1.2.1.3.5 Failure Path:** If the allocation fails (e.g., no duo available), the BE returns an error. The FE displays a **"Booking Failure"** message and the client remains on the **Book Now page** to retry or select a different slot.

##### 1.2.1.4 Session Maintenance
*   **1.2.1.4.1 Cancellation / End Booking:** Client or either assigned companion can cancel/end a booking at any time. On client cancellation, the client is shown a **review/confirmation page** before the cancellation is finalized. No time-based restrictions for now.
    *   **API:** Single cancel endpoint used across all phases: **[POST] /bookings/{id}/cancel**.
    *   **Roster Release:** On cancellation, the companion roster slot is **released** and becomes available for other bookings.
    *   **Companion Notification:** Companions are **notified** of the cancellation.
    *   **Redirect:** Client is redirected to the **home screen**.
*   **1.2.1.4.2 Identity:** The system uses the client's profile identity. The client sets a nickname during onboarding which is visible to the companions.
*   **1.2.1.4.3 Admin Logic:** Admin has the authority to cancel a client's booking (triggering a notification). Admin cannot "update" a client booking, only cancel.

##### 1.2.1.5 Accessibility & Localization
*   **1.2.1.5.1 Standards:** Initial implementation follows standard OS accessibility settings; no bespoke luxury-tier accessibility features are planned for Phase 1.
*   **1.2.1.5.2 Localization:** App UI and AI chat agent support **only English and Arabic** as of now.

#### 1.2.2 Functional (Companion) 
*   **1.2.2.1 Roster Booking:** Once the booking request is confirmed and companions are matched, the companion's **roster is booked** for that slot. The slot is no longer available for other bookings.
*   **1.2.2.2 Calendar & Blocking:** 
    *   **Logic:** The system **Auto-Assigns** the booking to the Duo. There is **no option to Decline** or Accept.
    *   **Block Duration:** The system blocks the Companion's calendar for the **2-hour session**.
    *   **No Immediate Notification:** The companion does **NOT** learn about the booking at this point. They only see a blocked slot on their calendar with no details.
*   **1.2.2.3 Detail Reveal (T−X Hours, currently 5 hours):**
    *   At **X hours** (configurable, currently set to **5 hours**) before the booking start, the companion is shown:
        *   **Time** of the booking.
        *   **Location** (venue) to go to for client matching.
        *   **Client Nickname**.
    *   No other client details are shared. This is done to **protect client privacy**.
*   **1.2.2.4 Companion Identity:** Companions are referred to by the **display name** they set during onboarding. This can be a nickname but is **never their real name**.

#### 1.2.3 Allocation Engine
*   **1.2.3.1 Booking Status Lifecycle:**
    *   `CONFIRMED` — Booking created, companions assigned, waiting for session day.
    *   `ACTIVE` — Session in progress (client-companion match completed, clock running).
    *   `COMPLETED` — Session ended normally (duration reached).
    *   `CANCELLED` — Booking cancelled before or during session (by client, companion, or admin).
*   **1.2.3.2 Strategy:** "First-Available" — the system picks the first available companion duo from the roster. Future iterations may add weighting based on ratings.
*   **1.2.3.3 Duo Integrity:** A booking is only valid if both a Captain and a Vice Captain are available for the full duration.
*   **1.2.3.4 Venue-Based Roster:** The roster is **venue-based** (not geo-location-based). Each venue has a roster that tracks companion availability. For a selected venue, the system checks the roster to determine available slots.
*   **1.2.3.5 Matching Flow:** The detailed matching flow is covered in **[1.4 Matching Flow](1.4_Matching_Flow.md)**.

#### 1.2.4 Admin
*   **1.2.4.1 Cancel:** Admin can cancel any booking (triggers a notification to the client).
*   **1.2.4.2 Reassign (Deferred / V2):** Admin can reassign one or both companions on a confirmed booking via selective allocation (providing `captain_id` and/or `vice_captain_id`) without cancelling the booking.
